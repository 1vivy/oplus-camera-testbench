# S8 — 8K configure_streams 0x80a9 −38

**PATH (crash-site → root):** crash at **C5/feature2** (`control/C5-camx-chi-feature2.md`) — the 8K (7680×4320, op_mode `0x80a9`) EISv2 node resolves 2-in/0-out → NULL pipeline → `configure_streams` returns −38. Two **parallel root candidates** feed it, neither a blob edit (both upstream blobs byte-identical OOS↔LOS):
- **C3** (`control/C3-cameraserver-extimpl.md`, PROXIMATE-UPSTREAM): LOS lacks the Depth-2 `beforeConfigureStreamsLocked` StreamSet-mutation + `getExtensionOperatingMode` that retype/bind the 8K EIS output stream — root is missing facilitation E2 (call sites) + E1 (`libcsextimpl`, dropped `d654641`), not a libcameraservice edit. **Edge:** C3 →(graph)→ C5.
- **D1** (`data/D1-gralloc-camxformat.md`, parallel root: Gralloc5 stream-usage resolution, doc-35 cand-a) — if the 100xx/Depth-2 path is answered yet the stream stays mis-typed, #8 falls to D1 usage-bit resolution (root then E4 namespace). **Edge:** C3 →(usage)→ D1.

**Decisive probe → verdict:** `tools/observability/r4-oem-transact/` (does LOS return `UNKNOWN_TRANSACTION` on OEM 100xx while SDK believes channel live) + `hook_eisv2_ports.js` (8K node in/out port count) settle which candidate is the root, per C3(c)/C3(f). If 100xx UNKNOWN **and** EISv2 lacks its output port → root = **C3/E2**; if 100xx answered but stream still mis-typed → root = **D1 (Gralloc5 usage)**. doc-48.

**STATUS:** characterization UNCHARACTERIZED / conviction OPEN — path only proposed end-to-end; runtime A/B is OOS↔LOS, **LOS-deferred** (this phase ends `conviction: SUPPORTED (LOS deferred)` once r4 + 8K port trace run on an unfrozen build; #8 capture currently gated behind the v19 freeze #1).

**RESOLVED (2026-06-24, on-device v2.1):** #8 root is **D1 (data-side dataspace), NOT C3/E2 Depth-2**. With frida `20_trace_ext_transact` armed, R4 Depth-2 (`beforeConfigureStreamsLocked`/`getExtensionOperatingMode`) fired **0×** yet 8K recorded cleanly (`7680×4320 fmt 0x22`, no `−38`) → the **C3/E2 candidate is disproven**. No LOS build ever returned `−38` (stock-trace-only); the real v1.4 8K failure was a `camera.oemlayer.healthmonitor` SIGABRT `ncsUnreleased 16` on a divergent IMPL_DEFINED dataspace (LOS `0x10c60000` BT2020 vs OOS `0x104` BT.709). Fixed **incidentally** by `getBlastSurfaceControl→null` (`android_hardware_oplus` `3d10b16`, authored for preview over-exposure), which drops the 8K record surface back to OOS-matching BT.709 — so the surviving root is **D1 dataspace via the EDR/BLAST path**, not Gralloc5 usage-bits and not C3/E2. Evidence: `reference/campaign/_los-v21-notes/8K-R4-finding.md`; `reference/_golden-oos-V16.1.0/campaign/video8k/frida/hook_configure_streams.log` (OOS `0x104`) vs `reference/campaign/video8k/frida/hook_configure_streams.log` (LOS v1.4 `0x10c60000`).

**UPDATE (v2.2, on-device 2026-06-25) — a SECOND, distinct 8K break appeared once R4 shipped, now FIXED.**
The "R4 fires 0×" finding above was **v2.1, where R4 was not yet in the build**. v2.2 baked the R4 configure
hooks (`ff7a3713a`) into the **statically-linked `/system/bin/cameraserver` binary**, so they now fire — and
`getExtensionOperatingMode` **clobbered the op_mode**: its trailing `int` arg is the *fallback op_mode* echoed
when the OEM override-tag is absent (always, on LOS — alias table dropped in R2), but the wiring passed
`atoi(mId)` (camId). So it returned the camera id → `mOperatingMode` `0x80a9`→`0x2` (8K cam2; selfie cam1→`0x1`)
→ `endConfigure: Unsupported set of inputs/outputs` → no preview. This **is** the C3/E2 (Depth-2) path — but a
*mis-wired-arg* mechanism, distinct from the original "2-in/0-out NULL-pipeline −38" theory (still unobserved on
LOS). **FIX:** `frameworks/av` `a536f0a481` — pass `mOperatingMode` (not camId) + `extMode >= 0x8000` guard;
rebuilt into the **cameraserver binary** (NOT the `.so` — it's inert, see `re-notes/cameraserver-static-link-build-traps.md`).
**Verified: 8K RECORDS 7680×4320 `.mp4`** (frida: getExtensionOperatingMode→`0x80a9` no override; CamX
`m_operationMode IS 0x80a9`; 0 configure failures). The D1 dataspace fix (`getBlastSurfaceControl→null`, `3d10b16`)
still holds and is **load-bearing for 8K** — authoring R3 (which needs `getBlastSurfaceControl`→REAL) would
re-break 8K (dataspace `0x10c60000`); treat R3 + the 8K dataspace as one problem.
