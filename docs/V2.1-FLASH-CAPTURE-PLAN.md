## ITERATIVE OVERLAY BRINGUP — R4 (Depth-2 hooks)
> **R4 configure hooks are now WIRED** (frameworks_av `ff7a3713a`, on top of the v2.1 R2+C1 zip):
> `getExtensionOperatingMode` + `beforeConfigureStreamsLocked` behind `isLoaded()` (no auth gate).
> Compiles (`mka libcameraservice` exit 0). The v2.1 **zip does NOT include R4** — overlay it: rebuild
> `libcameraservice` from `cam-final` (`mka libcameraservice`) → push `out/.../system/lib64/libcameraservice.so`
> to the flashed unit (or `mka bacon` for a v2.2 zip) → re-run the r4 probe. `afterConfigureStreamsLocked`
> + the exact hook args are still flash-to-confirm.

The honest next step after v2.1 confirms R2 loads. **R4 RE is done** (see
`docs/re-notes/oem-ext-depth2-lifecycle-RE.md`): the OOS root function `Camera3Device::configureStreamsLocked`
dispatches the ext via `getInstance()` → function-table → `blr table[slot]`, **gated by an "ext-enabled"
flag** at `device+0x3b4`, at 3 sites (op_mode override + before/after configure). The lifecycle flow + the
other roots (connect/request/result/disconnect) are mapped from the `getInstance` callers.

Bringup loop (overlay the wiring incrementally, re-capture each step):
1. Confirm with the r4 probe **which** Depth-2 hooks the app actually needs for the target mode (8K → `beforeConfigureStreamsLocked` + `getExtensionOperatingMode`).
2. Author the call site in OUR `Camera3Device::configureStreamsLocked` (before stream build) behind an
   ext-enabled gate; resolve the hook by dlsym (the `onTransact` precedent), args
   `(CameraMetadata sessionParams, op_mode, camera3::StreamSet&, String8 clientName)`.
3. `mka libcameraservice` → overlay-push the lib (or rebuild bacon) → re-run the r4 probe → confirm the
   hook now FIRES and the StreamSet is mutated for 8K.
4. Repeat per hook in lifecycle order.

## Known risks to watch at flash (from the static "full handler" audit)
- **sepolicy**: no explicit `libcsextimpl` rule found in our trees — cameraserver dlopen of a `system_ext`
  lib may hit an avc denial. Watch `logcat *:E` / `dmesg` for `avc: denied` on `cameraserver` ↔ `libcsextimpl`.
- **dlopen path**: factory uses the **relative** `"system_ext/lib64/libcsextimpl.so"` (resolves only if cwd is `/`).
  If `dlopen failed` in R2-1, hard-code `/system_ext/lib64/...` and rebuild.
- **Java tier-3 auth** (`OplusCameraManager.isAuthedClient`): verify it ships in the OCS SDK jar
  (`com.oplus.camera.unit.sdk.jar`); without it the SDK can't make the code-10004 auth call.
